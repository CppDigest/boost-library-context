# #323 - Tls 1.3 client does not report failed handshake when client certificate verification by server failed (Tls 1.2 client does) [Closed]

### Background  
We use boost beast, and create both clients and servers. We use them for testing that certain handshakes succeed or fail (depending on the configuration of the beast clients/servers) when connecting to our stack, or for simple Http requests and the respective responses (that we cannot trigger in our stack directly as a lot of it happens automatically).  
  
Doing so, we noticed a peculiar difference in the boost asio client behaviour, depending solely on whether one uses Tls 1.2 or 1.3.  
  
### Actual Issue  
When the server rejects the client's certificate, i.e. the client's certificate verification fails, a boost asio TLSv1.2 client will report an error from the handshake, specifically the errorcode will have the value: '336151608' and corresponding message 'tlsv1 alert internal error'.  
  
In contrast, a boost asio TLSv1.3 client will report no error from the handshake, i.e. an errorcode will have the value '0' and corresponding message 'The operation completed successfully'.  
  
### Version  
We are using:  
-  boost 1.71.0  
-  OpenSSL 1.1.1d  
  
### Steps necessary to reproduce the problem  
It can be reproduced using boost asio only.  
For this, you need a client and server.  
Configure both to verify their peers.  
Configure the server to have a verification callback that simply returns false (i.e. the verification of the client's certificate will fail in any case.)  
Configure the client to have a verification callback that returns true, i.e. the verification of the server certificate will succeed in any case.  
Configure both to use TLSv1.2 or TLSv1.3 respectively.  
  
### Expected Behavior  
When trying to connect to the server, the handshake fails since the server rejects the client's certificate.  
  
### Actual Behavior  
A TLSv1.2 client will behave as expected.  
A TLSv1.3 client will not realize a failed handshake.  
  
### Resources  
This [MWE](https://github.com/test-scenarios/beast-issue-1817) was created to recreate the issue.  
  
The [issue](https://github.com/boostorg/beast/issues/1817) was first posted in the beast repo before realizing that there is no involvement of boost beast in the actual issue.
