## Comment 2

> Username: pdimov

> Url: https://github.com/boostorg/atomic/issues/51#issuecomment-958282182  

> If so, it sounds like there is a bug in the CMake config generator.  
  
Maybe, but it's not so simple. First, CMake PUBLIC is merely a shortcut that sets both its equivalent of `requirements` and `usage-requirements` (`INTERFACE` is `usage-requirements` and `PRIVATE` is `requirements`.) So to generate `PUBLIC` we'd need to somehow extract the things that are in both b2's `requirements` and `usage-requirements`. But in this case it doesn't matter, because `IMPORTED` targets don't support `PUBLIC` or `PRIVATE`, just `INTERFACE`.  
  
I no longer understand what `boost-install` does, if I ever did, but I suppose I just take `requirements` and put that into `INTERFACE_LINK_LIBRARIES` (there's no other place to put it.) This works for e.g. Filesystem which links to both `bcrypt` (which it has in its `requirements`) and `synchronization` (which it picks up from Atomic.)  
  
It's still technically a bug, I suppose. It's odd for a library to link to something else just in its `usage-requirements` (that would be the b2 equivalent of a CMake compiled library having an `INTERFACE` dependency to another compiled library instead of `PUBLIC`) but it's technically legitimate.

## Comment 3

> Username: Lastique  
> Created at: 2021-11-02 22:45:03 UTC  
> Url: https://github.com/boostorg/atomic/issues/51#issuecomment-958318869  

> But in this case it doesn't matter, because `IMPORTED` targets don't support `PUBLIC` or `PRIVATE`, just `INTERFACE`.  
  
This is fine for shared libraries, but not for static. For shared libs, you only need to link with the library and whatever other dependencies that are needed just by its public headers (i.e. `usage-requirements` in b2 terms, `INTERFACE` in CMake terms). You don't need to link with `PRIVATE` dependencies (`requirements` in b2) as the shared library is already linked with them. For static libs, you also need to link with `PRIVATE` dependencies.  
  
If that's not possible to express in one target then we either need to create two sets of targets (for shared and static libs) or conditionally add dependencies based on `BUILD_SHARED_LIBS` or `BOOST_ALL_DYN_LINK` or something.
