Ouch, I see.  
  
You're probably already aware, but the Net TS replaces the allocation hooks with a single `get_associated_allocator(handler)` function. This should have the benefit that the forwarding chain only has to be ascended a single time, at operation initiation time, rather than once for every allocation and once for every deallocation. This might mitigate the problem, because it ideally would return an allocator that won't depend on any individual handler participating in the forwarding chain, save perhaps the top-most.

> Username: harrishancock

> Updated_at: 2017-01-08 15:15:26 UTC  
> Url: https://github.com/boostorg/beast/pull/218#discussion_r94708151  

Consider implementing this function as `release_handler()(std::forward<Args>(args)...)`. The only difference between the two expressions that I can think of is that `invoke` leaves the handler in place, while `release_handler()(...)` would zap the handler by moving it into its return value, but since both functions first destroy/deallocate `data`, I don't think that'd be a problem. I suppose the difference between the two could be removed by making `release_handler` return an rvalue reference, allowing the recipient of the ref to make the decision to zap or not, but I'm not sure how palatable/correct that is.
