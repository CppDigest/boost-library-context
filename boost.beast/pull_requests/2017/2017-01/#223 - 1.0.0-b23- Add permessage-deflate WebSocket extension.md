> Username: HowardHinnant  
> Created_at: 2017-01-11 18:50:56 UTC

> Url: https://github.com/boostorg/beast/pull/223#discussion_r95645118  

The use of `break` and `return` do the same thing, but spelling them differently like this doesn't give me confidence that the code is doing what the author intended.  I recommend using `break` in both places or `return` in both places.  
  
Also it might be nice to combine this like so:  
````  
       if (ec)  
       {  
             if( ec == zlib::error::need_buffers ||  
                 ec == zlib::error::end_of_stream)  
                 ec = {};  
             return;  
       }  
````

> Username: vinniefalco  
> Created_at: 2017-01-11 18:59:14 UTC  
> Updated_at: 2017-01-12 19:45:57 UTC  
> Url: https://github.com/boostorg/beast/pull/223#discussion_r95646850  

The idiom used throughout the library is as follows:  
1. Errors cause a function to terminate control flow immediately via `return`  
2. For functions returning `void` which complete successfully, the flow of control always runs to the end of the function  
3. For functions returning other than `void` which complete successfully, the flow of control always runs to the function epilogue where the return value is calculated.  
  
I preferred applying this style consistently. If I was to change it in the manner suggested, it would stand out as the only function that is written differently. It might not look like it matters much in this particular function, but its especially important in the invocation members for composed operations (or else incorrect behavior results) - so I preserved the style throughout.
