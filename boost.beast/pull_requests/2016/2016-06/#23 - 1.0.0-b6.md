## Comment 6

> Username: HowardHinnant  
> Created_at: 2016-06-03 18:58:49 UTC  
> Url: https://github.com/boostorg/beast/pull/23#issuecomment-223664408  

There's a growing danger of deriving from `detail::types`.  When considering ADL the compiler makes a list of associated namespaces within which it will search for overloads.  One of the associated namespaces is that of a base class (even if privately derived).  So when you have `beast::X` derive (even privately) from `beast::detail::Y`, `beast::detail` becomes an associated namespace of `beast::X` (an implementation detail leakage).  
  
This has the potential to cause either compile-time errors, or in extreme cases, even run time errors.  Here's a contrived demo:  
  
```  
namespace beast {  
namespace detail {  
  
template <class T>  
void  
tolower(T);  
  
struct base {};  
  
}  // detail  
  
struct derived : private detail::base  
{  
    operator long() const {return 0;}  
};  
  
}  // beast  
  
// client code  
  
#include <cctype>  
  
int  
main()  
{  
    beast::derived d;  
    using namespace std;  
    tolower(d);  
}  
```  
  
The client intends to call `std::tolower()` from `<cctype>`, but ends up calling `beast::detail::tolower` because `beast::detail` is an associated namespace of `beast::derived`, and `beast::detail::tolower`  turns out to be a better match than `std::tolower`.  
  
An alternative design using private data members instead of private inheritance avoids this issue:  
  
```  
struct derived  
{  
private:  
    detail::base data_;  
public:  
    operator long() const {return 0;}  
};  
```
