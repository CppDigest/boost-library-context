The way the `flat_stream` works is that on writes, when a buffer sequence has length greater than one (i.e. multiple buffers in the sequence, which is what Beast emits when serializing HTTP headers), it allocates memory and copies the data into a single buffer which is then sent. It has a limit to how many kilobytes it will allocate. Beyond this limit, it breaks the sequence down into pieces where each piece is no larger than the limit. It sends big blocks of memory individually. This is designed to reduce the number of calls to the `ssl::stream` write function.  
  
>Apologies in advance, if I've done something silly.

## Comment 11

> Username: vinniefalco  
> Created_at: 2018-05-01 18:08:26 UTC  
> Updated_at: 2018-05-01 18:10:38 UTC  
> Url: https://github.com/boostorg/beast/pull/1110#issuecomment-385743715  

What dominates performance is the **number** of calls into the kernel per unit time, which will be equal to the number of calls to the underlying `boost::asio::ip::tcp::socket` `write_some` or `async_write_some` operations. If you measure the number of calls for the same workload you should see that it is less than or equal to the number of calls in the previous implementation. The cost of the kernel transition whcih invokes underlying network driver is so significant that it makes other considerations, such as an additional dynamic memory allocation or added work in `memcpy` less relevant.  
  
This leads to an obvious heuristic for optimizing network programs: minimize the number of calls to read and write operations (but always measure).
