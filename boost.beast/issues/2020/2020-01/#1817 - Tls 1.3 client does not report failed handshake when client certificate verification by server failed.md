# #1817 - Tls 1.3 client does not report failed handshake when client certificate verification by server failed [Closed]

### Background  
  
We use boost beast, and create both clients and servers. We use them for testing that certain handshakes succeed or fail (depending on the configuration of the beast clients/servers) when connecting to our stack, or for simple requests and the respective responses (that we cannot trigger in our stack directly as a lot of it happens automatically).  
  
Doing so, we noticed a peculiar difference in the boost beast client behaviour, depending solely on whether one uses Tls 1.2 or 1.3.  
  
### Actual Issue  
  
When the server rejects the client's certificate, i.e. the client's certificate verification fails, a boost beast TLSv1.2 client will report an error from the handshake, specifically the errorcode will have the value: '336151608' and corresponding message 'tlsv1 alert internal error'.  
  
In contrast, a boost beast TLSv1.3 client will report no error from the handshake, i.e. an errorcode will have the value '0' and corresponding message 'The operation completed successfully'.  
  
### Version of Beast  
  
We are using boost 1.71.0  
  
### Steps necessary to reproduce the problem  
  
It can be reproduced using boost beast only.  
For this, you need a beast client and server running in different threads/processes (when using synchronous calls) or use asynchronous calls.  
Configure both to verify their peers.  
Configure the beast server to have a verification callback that simply returns false (i.e. the verification of the client's certificate will fail in any case.)  
Configure both to use TLSv1.2 or TLSv1.3 respectively.  
  
See attached file (the code is not running as is since you need to configure your keys and certificates)  
[TestTlsVerifyIssue.zip](https://github.com/boostorg/beast/files/4102435/TestTlsVerifyIssue.zip)
