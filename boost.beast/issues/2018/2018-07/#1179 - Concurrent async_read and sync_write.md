I have a question regarding concurrent async_read and async_write. According to  
[https://www.boost.org/doc/libs/1_67_0/libs/beast/doc/html/beast/using_websocket/notes.html#beast.using_websocket.notes.thread_safety](https://www.boost.org/doc/libs/1_67_0/libs/beast/doc/html/beast/using_websocket/notes.html#beast.using_websocket.notes.thread_safety), in multithreaded evironment, we can have at max 1 async_read and at 1 async_write. However, I am having issue with such concurrent async_read and async_write. I am wondering if it is feasible to have concurrent async_read and async_write.  
  
### Version of Beast  
Boost 1.67.0  
  
### Steps necessary to reproduce the problem  
Modify websocket_client_async_ssl.cpp to have at max 1 async_read and at max 1 async_write at a given time  
  
### All relevant compiler information  
g++ (Ubuntu 7.3.0-16ubuntu3) on Ubuntu 18.04  
openssl-1.0.2-n  
  
### Details  
I modified websocket_client_async_ssl.cpp example so that I can have concurrent async_read and async_write. The program ran for a while and crashed. I set breakpoint just before the crash and collected debug information. I attached it at the end of this section. As you can see, thread 1 and thread 3 accessed same SSL data structure. This eventually corrupted peer_b->len and caused the program crash. Thread 1 was doing a read operation and thread 3 was doing a write operation.
