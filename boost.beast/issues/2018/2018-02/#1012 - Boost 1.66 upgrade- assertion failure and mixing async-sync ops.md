I recently upgraded from Beast beta 0.69 to official Boost 1.66. (Belated congratz to Vinnie :-)  
  
I have a SSL websocket client test application for sending various test traffic to my SSL websocket server application. This test client can create multiple concurrent websocket connections and multiple worker threads  all managed under a single asio::io_service.  Each connection session uses a mix of asynchronous reads and synchronous writes (including synchronous websocket::stream::close()). The async read callbacks are wrapped by a strand. The synchronous writes/close are executed from within strand-wrapped callbacks or strand-posted functions. There is at most one active async_read() at a time.  
  
This test client worked fine with Beast 0.69. After upgrading to Boost 1.66, I noticed that when these client websocket connections attempt to initiate the close:  
1) websocket::stream::close() always produced the error code websocket::error::failed  
("boost.beast.websocket: WebSocket connection failed due to a protocol violation")  
2) Sometimes one or more of the connection sessions would assert fail at:  
beast/websocket/impl/read.ipp:576  
Assertion `ws_.status_ == status::open' failed.  
  
  
As an experiment, I changed close() to async_close() and then I noticed:  
1) The error code websocket::error::failed went away. async_close's callback always receives a success status.  
2) The read.ipp:576 "ws_.status_ == status::open" assertion failure went away.  
3) A new assertion failure appeared, but much less frequently:  
beast/websocket/impl/read.ipp:217  
Assertion `ws_.rd_block_' failed.  
The async_read() callback handler was not reached, so I'm guessing this asserted while processing the peer close frame response from the server.  
  
  
Questions:  
  
1) Is it incorrect to mix async and sync calls?   
  
I understand that there can be no more than one active async read, write, ping, close op without an intervening corresponding callback.  I also understand that websocket objects are not thread safe so some kind of synchronization control (like an explicit strand) is needed when using multiple threads.  
  
I think I'm following these rules...   But perhaps if websocket::stream::close() is internally performing async ops on the websocket, then these async ops might get executed on another thread without strand protection. This could race with my earlier call to async_read() if the timing is unlucky and end up with two threads modifying the same websocket.  
  
But the docs for close() says it's implemented with the next layer's write_some() which are blocking calls.  
  
Not sure if this is my bug or Beast bug  
  
2) what might be causing the new assertion failure at read.ipp:217?

Since it's still a very intermittent problem, I sprinkled websocket/impl/read.ipp and close.ipp with generous amounts of debug messages so that I could see the code flow in normal and assertion failure scenarios.  
  
I think the intermittent assertion failure is caused by a race condition when a co-routine yields and context switches to the "wrong" (or unexpected) scheduled function.  
  
Test scenario:  
  
The test client establishes a SSL websocket with the server. The server sends a message back to the client. The client closes the connection after receiving the message.  
  
The problem happens after the close frame is sent. In read.ipp (looking at what's on "master" branch right now):  
https://github.com/boostorg/beast/blob/908f84cb3c023ed7c5bd19e2e5283d3218a6d820/include/boost/beast/websocket/impl/read.ipp#L248  
  
  
                if( ws_.paused_r_close_.maybe_invoke())  
                {  
                    // Suspend  
                    BOOST_ASSERT(ws_.rd_block_);  
                    goto do_suspend;  
                }  
  
  
The maybe_invoke() is called and that sends us to close.ipp inside close_op<>::operator(), where we resume right after the yielded d.ws.paused_r_close_.emplace():  
  
https://github.com/boostorg/beast/blob/908f84cb3c023ed7c5bd19e2e5283d3218a6d820/include/boost/beast/websocket/impl/close.ipp#L189  
  
  
        {  
            // Suspend  
            BOOST_ASSERT(d.ws.rd_block_ != d.tok);  
            BOOST_ASIO_CORO_YIELD  
            d.ws.paused_r_close_.emplace(std::move(*this));  
  
            //// resumed here  
  
            // Acquire the read block  
            BOOST_ASSERT(! d.ws.rd_block_);  
            d.ws.rd_block_ = d.tok;  
  
            // Resume  
            BOOST_ASIO_CORO_YIELD  
            boost::asio::post(  
                d.ws.get_executor(), std::move(*this));  
            BOOST_ASSERT(d.ws.rd_block_ == d.tok);  
  
            // Make sure the stream is open  
            BOOST_ASSERT(d.ws.status_ != status::open);  
            BOOST_ASSERT(d.ws.status_ != status::closed);  
            if( d.ws.status_ == status::failed)  
                goto upcall;  
  
            BOOST_ASSERT(! d.ws.rd_close_);  
        }  
  
  
We follow this flow to the next yield: post(d.ws.get_executor(), std::move(*this))  (where "this" is the close_op).  
  
Here's where things diverge.  Most of the time, the yield sends us back to read.ipp from earlier at the maybe_invoke(). Here, maybe_invoke() returns true and we reach the assert for ws_.rd_block_, which we acquired just earlier in close_op<>::operator(). All is well.  
  
However, in the intermittent scenario where this assertion failed, the yield did not return to read.ipp. Instead, it jumped back into close.ipp right after the post(d.ws.get_executor(), std::move(*this)) !  
  
Following this flow in close.ipp leads us to reading a response close frame from the server and then finally the teardown. Then we eventually context switch back to read.ipp after the maybe_invoke().  The assertion fails since rd_block_ isn't acquired (probably because it's all torn down)

## Comment 10

> Username: vinniefalco  
> Created at: 2018-02-17 17:12:02 UTC  
> Updated at: 2018-02-17 17:14:45 UTC  
> Url: https://github.com/boostorg/beast/issues/1012#issuecomment-366455732  

When a composed operation schedules its own handler for a completion, it must use the same executor as that used to invoke the final handler. In this case, the strand is your executor. Therefore, Beast's composed operations must use that executor when performing its own intermediate asynchronous operations.  
  
To preserve this invariant, Beast composed operations have as their associated executor, the same executor as that associated with the final completion handler. This is accomplished by providing the nested type `executor_type` and member function `get_executor`, here:  
  
https://github.com/boostorg/beast/blob/1a88cb147bf15541df632ca60f277a02068bae5b/include/boost/beast/websocket/impl/close.ipp#L85  
  
Note that Beast returns the executor associated with the final handler. If there is no associated executor then it defaults to `ws.get_executor()` which is the executor for the `io_context`. This scheme is described here:  
  
http://www.boost.org/doc/libs/1_66_0/doc/html/boost_asio/reference/associated_executor.html  
  
If this mechanism was broken, either in Beast or in Asio, it could result in the symptoms you describe, where the strand is bypassed for intermediate operations. Perhaps there is a bug in Beast, and the line should read:  
  
```  
post(this->get_executor(), std::move(*this));  
```  
  
or even more simply:  
  
```  
post(std::move(*this));  
```  
  
If you make this change, you have to make it everywhere that `post` is called...sorry for the trouble!

This is very old behavior, it is implemented via the "teardown" customization point. The WebSocket protocol specification requires that the connection is actively closed as part of the closing handshake. So Beast provides a way to make sure it happens for the streams it knows about (plain TCP/IP and SSL). And there is a way for user-defined streams to make sure it happens, by providing suitable overloads for `teardown` and `async_teardown`:  
http://www.boost.org/doc/libs/1_66_0/libs/beast/doc/html/beast/using_websocket/teardown.html

Starting from the chat example program, add a strand to each HTTP and websocket session, and then modify `websocket_session::send` to post itself to the strand if it is not running on it already:  
```  
void  
websocket_session::  
send(std::shared_ptr<std::string const> const& ss)  
{  
    if(! strand_.running_in_this_thread())  
        return net::post(ws_.get_executor(), net::bind_executor(  
            strand_,  
            std::bind(  
                &websocket::session::send,  
                shared_from_this(),  
                ss)));  
  
    // Always add to queue  
    queue_.push_back(ss);  
  
    // Are we already writing?  
    if(queue_.size() > 1)  
        return;  
  
    // We are not currently writing, so send this immediately  
    ws_.async_write(  
        net::buffer(*queue_.front()),  
        std::bind(  
            &websocket_session::on_write,  
            shared_from_this(),  
            std::placeholders::_1,  
            std::placeholders::_2));  
}  
```

class listener : public std::enable_shared_from_this<listener>  
{  
	tcp::acceptor acceptor_;  
	tcp::socket socket_;  
  
public:  
	listener(  
		net::io_context& ioc,  
		tcp::endpoint endpoint)  
		: acceptor_(ioc)  
		, socket_(ioc)  
	{  
		beast::error_code ec;  
  
		// Open the acceptor  
		acceptor_.open(endpoint.protocol(), ec);  
		if (ec)  
		{  
			fail(ec, "open");  
			return;  
		}  
  
		// Allow address reuse  
		acceptor_.set_option(net::socket_base::reuse_address(true), ec);  
		if (ec)  
		{  
			fail(ec, "set_option");  
			return;  
		}  
  
		// Bind to the server address  
		acceptor_.bind(endpoint, ec);  
		if (ec)  
		{  
			fail(ec, "bind");  
			return;  
		}  
  
		// Start listening for connections  
		acceptor_.listen(  
			net::socket_base::max_listen_connections, ec);  
		if (ec)  
		{  
			fail(ec, "listen");  
			return;  
		}  
	}  
  
	// Start accepting incoming connections  
	void  
		run()  
	{  
		if (!acceptor_.is_open())  
			return;  
		do_accept();  
	}  
  
	void  
		do_accept()  
	{  
		acceptor_.async_accept(  
			socket_,  
			std::bind(  
				&listener::on_accept,  
				shared_from_this(),  
				std::placeholders::_1));  
	}  
  
	void  
		on_accept(beast::error_code ec)  
	{  
		if (ec)  
		{  
			fail(ec, "accept");  
		}  
		else  
		{  
			// Create the session and run it  
			std::make_shared<session>(std::move(socket_))->run();  
		}  
  
		// Accept another connection  
		do_accept();  
	}  
};
