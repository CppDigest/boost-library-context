### Steps necessary to reproduce the problem  
I'm currently developing some generic code which can use any kind of stream as template parameter. My specific case is using `tcp::socket`, `ssl::stream` and `websocket::stream`.  
  
However, it seems that when the `asio::async_write()` free function is used for `websocket::stream` the program fails to compile. However, `asio::async_read()` is compiling fine.  
  
To fix that issue I basically created the following free function  
```c++  
template <typename StreamType, typename... ArgsType>  
constexpr void async_write(websocket::stream<StreamType>& stream, ArgsType&&... args)  
{  
    stream.async_write(std::forward<ArgsType>(args)...);  
}  
```  
  
which solves the issue perfectly, but it's weird that I need to do that, given that async_read is working just fine.

That is correct. A websocket stream does not model a socket stream. The message boundaries in a websocket specifically delimit variable length messages, whereas the asio socket stream types model a connection with a continuous stream of data with no frame or message boundaries.

## Comment 5

> Username: madmongo1  
> Created at: 2021-06-03 14:14:41 UTC  
> Url: https://github.com/boostorg/beast/issues/2252#issuecomment-853902980  

beast's async_write_some is there to write multiple parts of a single websocket message, rather than to write some part of a stream.  
  
We have been asked previously for an async_write which would work on a buffers sequence like asio's.  
  
The issue is that it would be ambiguous as to whether eache buffer would be written to a single message, or whether each buffer would form its own distinct message.  
  
The same ambiguity would exist on the read side.
