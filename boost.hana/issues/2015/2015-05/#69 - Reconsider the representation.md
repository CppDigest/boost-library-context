> Url: https://github.com/boostorg/hana/issues/69  

Right now, we use a different template specialization to store each different string. This could also work by using an anonymous type, but there seems to be a bug in Clang right now that makes this in some circumstances: http://llvm.org/bugs/show_bug.cgi?id=20625. Using an anonymous type could have compile-time performance benefits, so this avenue should be explored once the bug is fixed.

## Comment 4

> Username: ldionne  
> Created at: 2015-11-20 16:01:16 UTC  
> Url: https://github.com/boostorg/hana/issues/69#issuecomment-158442086  

> Can you explain the term "anonymous type"? Is it struct with no name, or a struct generated by a lambda?  
  
I mean any type that is not _well-defined_, for some definition of well-defined. :-) Basically, it could be a lambda type, or even a local struct defined inside a lambda:  
  
``` c++  
auto f = []{ struct local { /* ... */ };  return local{}; };  
using Local = decltype(f());  
```  
  
These _weird_ types have the property of sometimes making symbols shorter, because they "erase" information about template parameters that can sometimes grow long. However, I think this is not the case with Clang, where the name of lambda type actually contains the context in which it was instantiated.  
  
Reducing the length of symbols might (or not) be an interesting optimization to apply, which is essentially the question behind this issue.

Since this issue has been opened, I have come to prefer the representation of strings as a parameter pack of `chars` since it provides better debug-ability over anonymous types. Furthermore, I will argue that  
1. both approaches don't scale to very large sizes  
2. the metaprogramming required to deal with such large strings would probably explode compile-times way before the string representation does  
  
Hence, I'll close this and keep the current representation. Overall, this issue will be better addressed by a language feature, which we are exploring as part of solving compile-time programming.  
  
__If you have a use case where the current representation of strings is a problem, let me know and we might revert this decision, but otherwise I would say you need to deal with it differently, e.g. by anonymizing the strings yourself.__
