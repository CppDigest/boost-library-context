Download and unzip a copy of boost 1.90.0 from https://archives.boost.io/release/1.90.0/source/  
  
```  
cd boost_1_90_0/libs/unordered/doc/html  
ls -al  
cd boost_1_90_0/libs/unordered/doc/html/unordered  
ls -al  
```  
The same files exist in both doc/html and doc/html/unordered.     
Where should they go...    
  
It seems like doc/html would be enough, and the last extra "unordered/" is superfluous.  Just put documentation in "doc/html" (unless there are other reasons. maybe there are.)    
  
A boost library is an example for other libraries. `boostorg/openmethod` now has the same issue. both should be fixed.

Thanks for the heads up. Turns out the copy of files from `html/unordered` to `html` is spurious and it's done here:  
  
https://github.com/boostorg/unordered/blob/1a034bfaddd1e7ebcafd730f7bde019e6a84b00d/doc/Jamfile.v2#L10-L14  
  
Changing that to  
  
```  
install install  
  : files-to-install  
  : <location>html  
    <install-source-root>html  
  ;  
```  
  
fixes the problem, but before going and doing the fix I'd like to ask @grisumbras, who I think is the original authour of this script, what the intent of this copying phase was and whether we could get rid of it altogether (including the `delayed-glob` rule).

## Comment 4

> Username: grisumbras  
> Created at: 2026-01-01 19:41:21 UTC  
> Url: https://github.com/boostorg/unordered/issues/330#issuecomment-3704043286  

The intent of that target is to run a glob _after_ the script that creates files to install has run. This is necessary to use b2's `install` rule on those files.  
  
But given that the files are generated by a bash script anyway, I recommend to just move the file copying/moving action there (or into a separate bash script).
