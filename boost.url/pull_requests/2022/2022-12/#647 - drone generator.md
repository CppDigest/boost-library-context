## Comment 22

> Username: sdarwin  
> Created_at: 2022-12-27 14:46:17 UTC  
> Url: https://github.com/boostorg/url/pull/647#issuecomment-1365953446  

The previous .drone.star style was about 100 lines of code, and very easy to parse and understand. The new one is 551 lines of code so far, and the logic for one job is spread across all the functions.  Maintaining or updating it with additional CI tests seems more challenging. I did like the old matrix where you could add a new test, and review all the relevant settings on one single line of job definition.  
  
In terms of the new design, what if the "generate" function could take an argument skiplist=[] which is an array, and the skiplist would allow you to skip any arbitrary subset of jobs, such gcc-8 or clang-15 or docs or tsan or anything. If _any_ job whatsoever was failing, you could add it to the skiplist for the time being, and it wouldn't cause CI to fail.  A skiplist could even potentially replace "docs=True, asan=True, tsan=True, etc etc" since you'd just add "docs" to the skiplist.  
  
Taking the skiplist design to it's extreme, the example .drone.star would look more like:  
  
```  
return generate()  
```  
  
By default it would build _everything_.  
  
and then you could customize it slightly:  
  
```  
return generate(skiplist=["clang-16"])  
```  
  
Consider that the goal is to update all cppalliance drone files from a central location. Let's say there's a new architecture such as s390x. Thanks for having a look, Sam.  
  
Yes. I agree the previous .drone.star file was shorter and easier to read. Scrolling through the jobs and knowing exactly what parameters each of them is using is quite useful.  
  
I think what's happening is these two approaches achieve different goals in terms of logic and use cases. In terms of logic, in both C++ and Starlark, it's reasonable to expect more lines of code whenever the user pushes the complexity to an external function because the external function is doing more than listing configurations. It implicitly lists all available configurations, describes their rationale, parses the user requirements, and matches them.  
  
They are also different in terms of use cases. For instance,   
  
- I think Peter doesn't use `functions.star` and he might usually have 200-300 lines of code.   
- The current code here is 100 lines of code but depends on `functions.star`, which is 283 lines of code. However, to be honest, I never cared to know this information until one minute ago. Complexity apart, if at least 2 repositories use `functions.star`, it's already worth it (283 + 100 x 2 < 300 x 2).   
- So from the perspective of the user, this is going from 100 lines of code to 3 lines of code, and it's globally worth it as soon as 2-3 repos use it.  
  
There's also the safety aspect since Vinnie's initial complaint was that he hadn't included the newly available job in one of his repositories and ended up not catching an error he could have.   
  
Regarding the design: yes. a skip list is worth considering. In practice, I don't think this blacklist is ever going to be empty though because we always need to skip old compilers and standards, and it might end up like an inverted version of the whitelist.   
  
```python  
def main(ctx):  
    return generate(skiplist=['gcc <4.8', 'clang <3.8', 'msvc <14.1'], skipcxx='<11')  
```  
  
The user might implicitly assume an invalid compiler will never be available for their library and rely on that informal agreement but that looks like a bad design from both parts. In general, since we'll still need to describe positive or negative requirements for the compilers, I think we will have the same pros and cons as we generally have for any whitelist/blacklist comparison:   
  
- Whitelists  
    - Pros: safer and there is no need to constantly evaluate items  
    - Cons: inflexible and requires more maintenance if the ranges are not open-ended  
- Blacklists:  
    - Pros: More flexible and easier to maintain  
    - Cons:   
        - Less secure as a newly available old compiler could break things   
        - Requires more decision-making  
  
However, one idea I had but didn't explore was separating the architectural information from the compilers. There would be no `arrm64-gcc`, ... anymore. Something like:  
  
```python  
def main(ctx):  
    return generate(['gcc >=4.8', 'clang >=3.8', 'msvc >=14.1'], ['x86', 's390x', 'arm64'], '>=11')  
```  
  
The reason I decided not to explore this before is that we just have a possible configuration per special architecture, and running all compilers again in another architecture looks like a waste of resources once the architecture has been validated.  
On the other hand, this could make the whitelist/blacklist discussion a little less relevant by making the generate function already shorter. One other possibility is to have functions to generate all compilers or architectures available but I'd have to think about that.
