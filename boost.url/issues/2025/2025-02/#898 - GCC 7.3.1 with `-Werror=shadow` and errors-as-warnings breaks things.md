I just revisited this issue.   
  
`-Werror=shadow` and `-Werror=conversion` **are** indeed pedantic warning.   
  
Boost libraries do not make an effort to ensure headers are free of pedantic warnings for many reasons:  
  
- The user can always include external headers without any warnings (pedantic or not), so warnings are a tool for the developer to find errors. Warnings that propagate this way are not errors in the library.   
- These pedantic warnings are full of false positives. They make code almost impossible to write and give back no benefit in terms of security (that's why they were made pedantic). For instance, in this case, I had changed ~400 LOC trying to get rid of this warning. It was very pedantic and useless (although the example you included seems reasonable), as it doesn't even allow variables inside a function definition to have the same name as a function in the class, and everything in a lambda is assumed to be ambiguous with the outer context. Not only this is useless in terms of security and expensive to maintain, but it creates situations where it's almost impossible to come up with a 4th, 5th, unambiguous name for the same thing, forcing some situations where code is actually less safe.  
- Even if we fix these warnings, other dependencies will trigger the same pedantic warnings and we'll have achieved nothing. For instance, after fixing all `shadow` warnings in Boost.URL, what we got is not headers without warnings. In this case, we just got headers with warnings from boost/filesystem/path.hpp:421:58.   
  
So after experimenting a lot of these flags and reaching these conclusions, I'll have to close this issue as not planned.
