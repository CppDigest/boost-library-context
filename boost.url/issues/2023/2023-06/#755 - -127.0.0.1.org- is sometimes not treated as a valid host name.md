# #755 - "127.0.0.1.org" is sometimes not treated as a valid host name [Closed]

> Username: Marc-Aldorasi-Imprivata  
> Created at: 2023-06-13 17:56:05 UTC  
> Updated at: 2023-07-18 21:50:45 UTC  
> Closed at: 2023-07-18 21:50:45 UTC  
> Url: https://github.com/boostorg/url/issues/755  

```c++  
boost::url u1;  
u1.set_scheme("http");  
u1.set_host("127.0.0.1.org");  
auto u1_str = u1.buffer(); // "http://127.0.0.1.org"  
auto u1_type = u1.host_type(); // host_type::name  
boost::url u2(u1_str); // Throws  
auto u2_type = u2.host_type(); // host_type::ipv4  
assert(u1_type == u2_type); // Fails  
```  
  
When using boost 1.82, the above code fails when constructing `u2` by throwing an exception with the string "leftover [boost.url.grammar:4]".  When using boost 1.81, constructing `u2` succeeds but interprets the host as an ipv4 address, which triggers the assert.  "127.0.0.1.org" is a valid host name: none of the characters in it are required to be percent-encoded by RFC 3986, RFC 1123 (which defines DNS names) says  
  
> a segment of a host domain name is now allowed to begin with a digit and could legally be entirely numeric  
  
, and fully numeric segments do occur in real life (for example, https://01.org is a real website).  Boost.URL should consistently treat "127.0.0.1.org" as a valid host name instead of throwing or treating it like an ipv4 address.
