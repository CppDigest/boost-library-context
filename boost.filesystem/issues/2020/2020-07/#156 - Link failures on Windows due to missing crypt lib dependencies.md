> Url: https://github.com/boostorg/filesystem/issues/156#issuecomment-664853944  

> Is it possible to keep the MSVC auto link statements?  
  
Technically, it's possible, but I don't want to do this. It is an MSVC-specific solution, which will not work when auto-linking is not supported, meaning that the build system based linking must still be implemented. I'd rather have it in a single place and work the same way everywhere.  
  
The dependencies are selected and declared in the build system - both Boost.Build and CMake. Consumers importing those build systems will keep linking as they did before.  
  
Consumers not importing the library build systems may be affected, if they link to a static library. They will have to reproduce the same checks we do in our scripts in their build system or just hardcode the required libraries. Or just switch to a shared library of Boost.Filesystem, which already has everything linked.  
  
I don't know how CMake FindBoost config files are generated. If they don't include library dependencies then I would say they should be updated to include those. If there is a way to do that from Boost.Filesystem side, I'm not aware of it. @pdimov Could you comment on this?

My point is: Users usually use the system provided Boost (through packages) or build it with b2 as building Boost with CMake is not yet working     
The installed Boost will have no mention what library was used to build Boost and hence which library the user needs to link because B2 doesn't write this information out (CMake does)  
  
So now something that worked before is broken and users have to do quite some work to workaround this as "reproduce the same checks we do in our scripts in their build system" may not be entirely trivial. Maybe it is even wrong if the Boost build environment and their environment is slightly different. (Not sure here, but maybe in the latter the library is found while it wasn't when Boost was build or the other way round) So I'm quite certain users will be confused and will complain once this is released.  
  
While I agree with that this is  a "MSVC-specific solution, which will not work when auto-linking is not supported,", MSVC is still the most widely used compiler on Windows. (Does clang-cl support this too?`I'd guess yes) So for the majority of the users this solution works. Windows users usually don't use shared libs either because that is a huge PITA, so they likely use static Boost libs  
  
The (upstream) CMake FindBoost script would need to add this check too, but the CMake people usually resist and say that the library project (here: Boost) should do that work as maintaining such a module outside a library is... hard.  
  
The Boost CMake Configs generated by B2 currently don't have the ability to declare dependencies which is a shortcoming which might not be solved before the upcoming release.  
  
So in order to not alienate the majority of Windows users of Boost.Filesystem with the upcoming release I'd strongly recommend adding those few lines back.
