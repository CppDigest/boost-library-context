# #21 - boost::multiprecision::int128_t gives unexpected results when exporting and importing bits [Closed]

> Username: cpcloud  
> Created at: 2017-04-09 18:09:30 UTC  
> Updated at: 2022-02-25 09:24:09 UTC  
> Closed at: 2017-05-31 18:32:27 UTC  
> Url: https://github.com/boostorg/multiprecision/issues/21  

When using `export_bits` with the number `2 ^ 127 - 1`, I cannot roundtrip convert between `int128_t` values and bytes generated by `export_bits` on that value.  
  
```cpp  
// largest positive number representable by two's complement 128-bit integer  
int128_t value("170141183460469231731687303715884105727");  
uint8_t stack_bytes[16] = {0};  
uint8_t* bytes = stack_bytes;  
export_bits(value, bytes, 8, false); // chunks of 8 bits, LSB first  
```  
  
If I then immediately do:  
  
```cpp  
int128_t imported_value;  
import_bits(imported_value, bytes, bytes + 16, 8, false);  
```  
  
The number I get back is `2 ^ 128 - 1`, with every bit in each underlying byte set to 1. I understand that boost is *not* using two's complement, but sign-magnitude.  
  
In either case, I would expect that `2 ^ 127 - 1` is represented by 127 bits set to 1 and the 128th bit set to 0 instead of all 128 bits set to 1.  
  
Am I misunderstanding something about the representation, or is this expected behavior?  
  
More generally, Is there a documented/sanctioned way to recreate an `int128_t` number from its underlying bits? I'm fine with handling the sign bit on my own.

> Username: jzmaddock  
> Created at: 2022-02-24 12:36:58 UTC  
> Url: https://github.com/boostorg/multiprecision/issues/21#issuecomment-1049817209  

This is expected, all cpp_int's are sign-magnitude representation, not 2's complement, so the maximum value in a uint256_t and an int256_t are the same and are 2^256 - 1.  
  
So.... you're doing nothing wrong, but this is intended behaviour, and these types are NOT bit-compatible with 2's complement built in integers.  For 99% of use cases this makes no difference, it's only when you start doing bit-fiddling on signed numbers (which is undefined behaviour anyway) that you see the difference.
