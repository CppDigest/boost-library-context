Here's a test case (simplified code from a Boost.Integer test):  
  
```  
#include <limits>  
#include <boost/multiprecision/cpp_int.hpp>  
  
template< typename T >  
inline int integer_log2_impl(T x)  
{  
    unsigned int n = std::numeric_limits< T >::digits / 2;  
  
    int result = 0;  
    while (x != 1)  
    {  
        T t(x >> n);  
        if (t)  
        {  
            result += static_cast< int >(n);  
            x = static_cast< T&& >(t);  
        }  
        n >>= 1u;  
    }  
  
    return result;  
}  
  
int main()  
{  
    return integer_log2_impl(boost::multiprecision::cpp_int(65536));  
}  
```

g++ -O3 -DNDEBUG -Wall -I. -std=gnu++14 -o mp_maybe_uninitialized mp_maybe_uninitialized.cpp

OK, a quick diversion first:  in your code, `numeric_limits<cpp_int>::digits` is `INT_MAX` which is a rather large number, also calling `integer_log2_impl(boost::multiprecision::cpp_int(65536) << 1000);` goes into an infinite loop with n going to zero and x != 1.  But I realise it may not be complete code.  
  
Now for the issue, and I'm not sure what to do about this - as GCC is both being very clever and quite dumb at the same time.  During the assignment variable t is first default initialized, and then assigned the expression template (inside its constructor).  The default initialization does not - as GCC notes - initialize the union member complained about, but does set member m_internal to true, so in the call to limbs(), m_internal is always true and m_data.ld.data is never returned.  GCC doesn't spot that bit :(  
  
The issue here is that m_data is a union with members la and ld, only one of which is ever active, as indicated by the flag m_internal.  Now, the thing is, the default initialization only initializes the first member of array la: because that's all that's needed, and all that will be accessed.  But it does mean that m_data.ld.data resides in uninitialised memory.  So... I can suppress the warning by initializing member la using array initialization syntax... but while that suppresses the warning it's worth noting that m_data.ld.data is still a garbage pointer because that union member is not active.  Plus there are other constructors where this trick won't work, and it's array-initializing data that doesn't need to be initialized.  
  
So.... what to do?  Any ideas anyone?  
  
BTW related issues crop up with various static analysers like clang-tidy as well.
